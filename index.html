rebase命令

之前在项目开发的时候，为了方便快捷，合并分支总是用merge命令，虽然用起来简单，不用过多考虑，但是会看历史操作，会发现commit很乱，特别是多人协作开发一个项目的时候。

而且会产生一些多余的commit比如上图中的⭕️标注的地方，为什么会产生呢。这和merge命令的机制有关

git会找到俩个要合并的分支的共同祖先，这里也就是C2,然后会把C2、C3、C5做一个新的提交

我们发现C5其时是不必要的，而且分支信息会很乱，长期下去，就会导致提交信息很乱。

我们看到master分支超前develop俩个版本，这个虽然可以实现，但我们通常不这么用，因为master是公公分支，上面会有很多人的提交，其他人还在原来的 master上开发，由于你使用了rebase移动了master，git 会认为你的主分支的历史与其他人的有分歧，会产生冲突。

而且rebase命令会生成新的哈希值，修改了原来的commit版本号，但内容没有变。如果要撤回：

rebase 版本

rebase命令除了可以进入分支rebase也可以进入某次commit进行rebase。可以合并提交，修改提交信息，丢弃不想要的提交等等

我们看到有一些操作commands：

  • pick 挑选当前的commit
  • reword 使用当前commit，但会修改commit message
  • edit 使用当前commit，允许你编辑文件，编辑后会并入当前的commit amend
  • squash 使用当前的commit，但会合并到之前的commit中去
  • fixup 和squash类似，但是会丢掉很次commit的commit message
  • drop 丢弃当前commit
